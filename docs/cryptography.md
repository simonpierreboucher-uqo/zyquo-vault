# Zyquo Vault — Cryptography

This document specifies the cryptographic design at the byte level. It must stay in sync with `Sources/ZyquoVaultCrypto/`; divergence is a reportable bug.

## Primitives (all established; nothing invented)

| Purpose | Primitive | Implementation |
|---|---|---|
| Password KDF | **Argon2id** v19 (0x13) | Vendored official reference implementation (ADR-0002), validated against the official known-answer vectors |
| Subkey derivation | **HKDF-SHA256** | CryptoKit, validated against RFC 5869 test case 1 |
| AEAD | **AES-256-GCM** | CryptoKit (hardware-accelerated) — ADR-0003 |
| Header field authentication | **HMAC-SHA256** | CryptoKit |
| Randomness | `SecRandomCopyBytes` | Wrapped in `SecureRandomSource` (injectable for tests) |

`SecRandomCopyBytes` lives in Security.framework but is a randomness API, not Keychain storage; its use is explicitly allowed under the Keychain prohibition.

## Key hierarchy

```
Master password + per-vault salt + Argon2id params
  → PKEK (32 bytes; exists only during unlock / password change / creation)
      → AEAD-unwraps the random 256-bit Vault Master Key (VMK) from the header
          → HKDF-SHA256 domain-separated subkeys (contexts below)
              → per-record / per-attachment random 256-bit DEKs (M2+)
                  → AES-256-GCM payloads
```

- The VMK is generated by the CSPRNG, never password-derived.
- Password verification **is** the authenticated unwrap of the VMK. Wrong password and corrupted header are deliberately indistinguishable at the API boundary (`CryptoError.invalidPasswordOrCorruptedVault`).
- Password change re-wraps the VMK (fresh salt, fresh PKEK) and never re-encrypts records.

### HKDF contexts (info strings, UTF-8)

```
zyquo-vault/v1/record-wrapping
zyquo-vault/v1/attachment-wrapping
zyquo-vault/v1/manifest-protection
zyquo-vault/v1/backup-protection
zyquo-vault/v1/search-session
zyquo-vault/v1/header-auth
```

HKDF salt = the 16 vault-UUID bytes (public; binds subkeys to one vault). Output = 32 bytes. A subkey is never reused across purposes.

## Argon2id parameters

- Salt: ≥ 16 bytes (≤ 64), unique per vault, from the CSPRNG.
- Output: 32 bytes (≤ 64).
- **Floors (enforced, cannot be parsed from a header below these):** 64 MiB memory, t = 3, p = 1.
- **Ceilings (DoS guard):** 4 GiB memory, t = 64, p = 8.
- Default baseline: 128 MiB, t = 3, p = 4. On-device calibration (`Argon2id.calibrate`) targets ~0.75 s per unlock at vault creation, growing memory first, then iterations, never below the floors.

## Canonical AAD encoding (v1)

Every ciphertext is bound to its context via this fixed 57-byte structure. All integers big-endian.

| Offset | Size | Field |
|---|---|---|
| 0 | 4 | magic `"ZQAD"` (5A 51 41 44) |
| 4 | 1 | AAD version = 0x01 |
| 5 | 16 | vault UUID (RFC 4122 byte order) |
| 21 | 16 | object UUID |
| 37 | 4 | object type (1 = header-VMK, 2 = record, 3 = attachment chunk, 4 = manifest, 5 = backup, 6 = recovery-VMK) |
| 41 | 4 | schema version |
| 45 | 8 | revision |
| 53 | 4 | algorithm (1 = AES-256-GCM) |

For the header-wrapped VMK: object UUID = vault UUID, object type = 1, schema version = header format version, revision = 0.

## AEAD rules

- Fresh 12-byte CSPRNG nonce per seal — never counter- or timestamp-derived.
- 16-byte GCM tag; wire form is `nonce ‖ ciphertext ‖ tag`.
- Keys must be exactly 256 bits.
- Any authentication failure throws; no partial plaintext is ever returned (CryptoKit verifies the tag before releasing plaintext).

## Header authentication model

Three complementary mechanisms (see `docs/vault-format.md` for the layout):

1. **KDF fields (salt, parameters):** tampering changes the derived PKEK, so the authenticated VMK unwrap fails. Fail closed.
2. **Vault UUID + format version:** bound as AAD to the wrapped VMK.
3. **Everything else (timestamps, flags, versions):** covered by HMAC-SHA256 over the serialized header body, keyed by the `header-auth` HKDF subkey, verified in constant time immediately after a successful unwrap.

A parser rejects malformed, truncated, oversized, unknown-flag, or DoS-parameter headers before any key derivation.

## Memory handling — honest limits

`SecureBytes` provides page-aligned, `mlock`ed (best-effort) storage, `memset_s` zeroization on `deinit`/`wipe()`, redacted descriptions, and scoped access. **Swift cannot guarantee perfect erasure**: ARC copies, `Data` bridging, `String` internals, and the optimizer can all create unreachable transient copies. The master password additionally exists briefly as a `String` at the UI boundary. These residual exposures are accepted and documented rather than hidden; the mitigations shorten the window, they do not close it.

## Test vectors

- Argon2id: the 8 official vectors from the reference implementation's `test.c` (v19), run against the vendored C code in `Tests/ZyquoVaultCryptoTests/Argon2idTests.swift`.
- HKDF-SHA256: RFC 5869 test case 1.
- AES-GCM: round-trip, tamper (ciphertext/tag), wrong-key, AAD-mismatch, nonce-freshness tests.
- Deterministic wrapped-VMK vectors for independent implementers will be added when the format freezes (pre-1.0), clearly marked unsuitable for real vaults.
